# debrief — 跑完一个长任务后，帮你搞清楚刚才发生了什么

一个为 OpenCode 构建的苏格拉底式提问 Agent，帮助你在 AI 辅助编码后重新建立对自己项目的理解。

## 产品洞察

### 问题的角度本身就是讲解

在传统的教育产品中，我们习惯于"先教后问"：AI 讲解概念，用户吸收知识。但我在使用 OpenCode 时发现了一种更有效的学习方式：**问题的角度本身就是讲解**。

当我让 AI 写了一堆代码后，如果导师问我"这个数据从哪来的？"、"你为什么用这个方案？"，我被迫停下来思考。而在思考的过程中，我重新理解了项目——不是我被动接收信息，而是主动构建认知。这种"适度的困难"（Productive Struggle）比直接得到答案的学习效果要好得多。

但"苏格拉底式提问"不等于"考试式提问"。如果 AI 只是问"这个数据流怎么工作的？"，用户只会觉得累——因为问题里没有信息，没有路径，只有压力。**好的苏格拉底式提问本身就是一个微课堂**：它包含用户可能没注意到的观察、一条可走的思考路径、以及一个邀请深入思考的提问。

### 为什么 OpenCode 的 Agent 机制天然适合教育场景

OpenCode 支持自定义 Agent——通过 `.opencode/agent/` 目录下的 markdown 文件即可注册。用户可以通过 Tab 键在不同 Agent 之间切换。

这意味着教育场景不需要任何额外的 UI 或交互设计：
- 编码时用默认的 build agent
- 想复盘时 Tab 切换到 debrief agent
- 复盘完了 Tab 切回去

### 两种范式对比

| 范式 | "AI 回答问题" | "AI 提出问题" |
|---|---|---|
| 用户角色 | 信息消费者 | 思考者 |
| 认知负担 | 低（直接接收） | 高（必须思考） |
| 学习效果 | 短期记忆好 | 深度理解好 |
| 适用场景 | 查询、快速实现 | 深度学习、概念掌握 |

### 为什么不做知识图谱

我一开始考虑过用知识图谱为用户建模知识边界，但很快意识到这太重了。一个简历项目不应该承担如此复杂的工程。

更关键的是，苏格拉底式提问的本质是**适度的困难**（Productive Struggle）——问题的难度应该刚好超出学生当前水平一点点。这需要的不是复杂的知识图谱，而是对上下文的敏锐理解。LLM 本身已经具备了这种能力，我们只需要通过良好的提示词引导它，而不是强行用知识图谱去限制它。

## 架构：Agent + Plugin

debrief 的架构非常简单，由两部分组成：

```
┌──────────────────────────────────────┐
│  .opencode/agent/debrief.md          │
│  苏格拉底式导师的完整 prompt          │
│  - 问题构建公式：观察 → 路径 → 提问   │
│  - 五种提问路径 + 自适应策略          │
│  - 启动时自动调用工具获取上下文        │
├──────────────────────────────────────┤
│  opencode-debrief (Plugin)            │
│  注册两个工具供 Agent 调用            │
│  - scan_project: 扫描项目文件结构     │
│  - analyze_session: 分析会话历史      │
└──────────────────────────────────────┘
```

**Agent 负责人格和提问策略**（纯 markdown），**Plugin 负责提供工具**（TypeScript）。两者通过 OpenCode 的工具调用机制连接。

这种分离意味着：
- 修改提问策略只需要改 markdown 文件，不需要重新 build
- 工具可以独立测试和扩展
- Agent prompt 对 HR 来说可读性极高——直接就能看到产品的核心设计

## 安装

### 1. 安装插件（提供工具）

在项目的 `opencode.json` 中添加插件引用：

```json
{
  "plugin": ["opencode-debrief"]
}
```

本地开发可以使用文件路径：

```json
{
  "plugin": ["file:///path/to/opencode-debrief/dist/index.js"]
}
```

### 2. 添加 Agent（提供人格）

在项目根目录创建 `.opencode/agent/debrief.md`，内容是苏格拉底式导师的完整 prompt。（参见本仓库的 `agent-example.md`）

## 使用方式

1. 在 OpenCode 中写代码（或让 AI 帮你写）
2. 觉得对项目不熟悉了？按 **Tab** 切换到 `debrief` agent
3. AI 会自动扫描项目和分析会话，然后开始提问——每个问题都包含具体观察和思考路径：
   - "我注意到你的项目里 scanner 和 history 是两个独立模块——scanner 读文件系统，history 查 API。你觉得为什么它们不是一个统一的'数据采集器'？"
   - "你的插件在用户切换到 debrief 时才扫描项目，而不是加载时就做。这个'懒加载'设计有什么权衡？"
4. 回答问题（AI 会根据你的回答调整后续提问方向）
5. 完成后，按 **Tab** 切回 build agent，继续编码

## 设计决策

### 为什么用 Agent 而不是 Command？

OpenCode 支持两种扩展方式：
- **Command**（`/xxx`）：一次性触发，执行完就结束
- **Agent**（Tab 切换）：持久的人格，可以持续对话

苏格拉底式提问是多轮对话——Agent 天然适配这个场景。Command 只能触发一次，无法维持"追问"的对话流。

### 为什么 Agent prompt 和 Plugin 分离？

- Agent prompt 是产品设计的核心——改它不需要重新编译
- Plugin 只提供基础设施（工具）——改它不影响提问策略
- HR 看到那个 markdown 文件就能直接理解产品理念，不需要读代码

### 为什么不做知识图谱？

- **太重**：简历项目不需要如此复杂的工程
- **不灵活**：知识图谱难以适应不同项目、不同学习阶段
- **过度工程化**：核心价值在"适度的困难"，不在精确的知识建模
- **LLM 已有理解能力**：通过良好的提示词即可引导 LLM 做自适应提问
